# RFC0001 - Simple Barcode Federation Protocol (SBFP)

## Abstract

This document specifies the Simple Barcode Federation Protocol (SBFP), a lightweight, decentralized protocol for sharing
product barcode information. SBFP defines a simple mechanism for server-to-server communication, including an automated
handshake for establishing peering relationships. To enhance network growth, it also specifies an optional but
recommended endpoint for public peer discovery. Nodes exchange data through direct HTTP POST requests, and data records
use JSON Schema for flexible, verifiable metadata.

## 1. Introduction

The lack of a comprehensive, open database for product barcode information hinders application development. The Simple
Barcode Federation Protocol (SBFP) solves this by defining a simple, decentralized way for independent servers to share
and synchronize barcode data.

Inspired by the federated design of protocols like ActivityPub, SBFP enables a resilient network where anyone can host a
Node and contribute data. The protocol uses a direct push model for data propagation and leverages JSON Schema to ensure
data records are structured, verifiable, and can be extended with rich metadata.

---

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL"
in this document are to be interpreted as described in BCP 14 [RFC2119][1] [RFC8174][2] when, and only when,
they appear in all capitals, as shown here.

- Node: A server instance participating in the federation.
- Record: A JSON object representing a single barcode and its metadata.
- Announcement: A JSON payload that wraps a Record to inform other Nodes of a Create or - Update event.
- Peer: Another Node in the federation that a Node communicates with.
- Federation: The overall network of interoperating Nodes.
- JSON Schema: A vocabulary for validating and annotating JSON documents.

---

## 3. Protocol Overview

SBFP uses a simple, direct push model for federation.

1. Discovery: A Node can discover a peer's announcement endpoint by querying a well-known URI on the peer's domain.

2. Peering: Nodes maintain a list of peers they wish to exchange data with. This can be managed manually or through
   automated discovery.

3. Publication: When a new barcode Record is created or updated on a Node, it wraps the Record in an Announcement
   payload.

4. Federation: The Node sends this Announcement payload via an HTTP POST request to the designated announcement endpoint
   of its peers.

5. Processing: Upon receiving an Announcement, a peer Node validates the payload and the Record within it. If valid, the
   Node updates its local database.

## 4. Detailed Specification

### 4.1. Node & Peer Discovery

Discovery begins with a standard discovery file. This file contains the node's essential endpoints and SHOULD include a
link to its public peer list but MAY omit it for privacy purposes.

#### Initial Discovery

A Node MUST provide a discovery file at `/.well-known/sbfp`. This file informs other nodes about its identity and where
to
send Announcements. The `peersEndpoint` property SHOULD be included to promote network discovery but MAY be omitted for
privacy reasons.

The discovery file contains the following properties:

- `nodeUrl` (string, REQUIRED): The canonical base URL for the node. This MUST be the root URL from which other
  endpoints are served.
- `nodeName` (string, REQUIRED): A human-readable name for the node
- `announcementEndpoint` (string, REQUIRED): The full URL where the node accepts incoming `Announcement` payloads.
- `peersEndpoint` (string, OPTIONAL): The full URL for the node's public peer list. This property SHOULD be included for
  network discovery but MAY be omitted for privacy.

**Example Discovery Request**:

**Request**: GET `https://barcodes.example.com/.well-known/sbfp`

**Response**: A JSON object containing the Node's metadata.

JSON

```json
{
    "nodeUrl": "https://barcodes.example.com",
    "nodeName": "RFC Barcode Example",
    "announcementEndpoint": "https://barcodes.example.com/sbfp-inbox",
    "peersEndpoint": "https://barcode.example.com/sbfp-peers"
}
```

#### Optional Peer List Endpoint

A Node MAY provide a `peersEndpoint` in it's discovery file. If provided a peers endpoint MAY be used to find additional
peers.

The response contains the following property:

- `peers` (array of strings, REQUIRED): An array where each string is the `nodeUrl` of a peer. The list MAY be empty if
  the node has no peers or chooses not to list them.

If a node does not wish to expose a peer list, the `peersEndpoint` property MUST be omitted from its discovery file. A
client attempting to access a non-existent `peerEndpoint` SHOULD handle a `404 Not Found` response gracefully.

**Example Peer Discovery Request**

**Request**: GET `https://barcodes.example.com/sbfp-peers`

**Response**: A JSON object containing the Node's peer list

JSON

```json
{
    "peers": [
        "https://barcodes.example.org",
        "https://barcodes.example.net"
    ]
}
```

### 4.2. Record Object Model

The Record is the fundamental data object in SBFP. It represents a single barcode.

The Record object has the following properties:

- `id` (string, REQUIRED): The ID contains the information contained on a barcode this MUST be all lowercase.
- `barcodeType` (string, REQUIRED): The symbology of the barcode (e.g., "UPC-A", "EAN-13", "QR_CODE").
- `metadataSchema` (string, REQUIRED): The full URI of the JSON Schema used to validate the metadata object. This makes
  the record self-describing.
- `metadata` (object, REQUIRED): An object containing type-specific information about the item. The structure of this
  object MUST conform to the schema defined by metadataSchema.

**Example Record Object**:

JSON

```json
{
    "id": "012345678912",
    "barcodeType": "UPC-A",
    "metadataSchema": "<placeholder>/schemas/types/food.json",
    "metadata": {
        "servingSize": "40g",
        "calories": 150,
        "ingredients": [
            "Whole Grain Rolled Oats"
        ]
    }
}
```

### 4.3. Announcement Payload

An `Announcement` is the envelope used to push events between nodes.

The base Announcement object has the following properties:

- `action` (string, REQUIRED): Defines the purpose of the announcement.
- `sourceNode` (string, REQUIRED): The nodeUrl of the node sending the announcement.

#### Data Announcement Actions

- `Create`: Announces a new `Record`. MUST include a `record` property
- `Update`: Announces a change to an existing `Record`. MUST include a `record` property. The `id` of the record MUST
  match an existing record.

**Example Announcement Payload**:

JSON

```json
{
    "action": "Create",
    "sourceNode": "https://barcodes.example.com",
    "record": {
        "...RecordObjectfrom": "4.2..."
    }
}
```

#### Peering Announcement Actions

These actions manage the peering handshake and MUST NOT contain a `record` property.

- `PeerRequest`: Initiates a peering request.
- `PeerApprove`: Approves a peering request. MAY contain additional information, such an an API key for the requester to
  use.
- `PeerReject`: Rejects a peering request. SHOULD include a `message` property explaining the reason.
- `PeerPending`: Informs the requester that their request requires manual approval.

**Example Peer Request Payload**:

```json
{
    "action": "PeerRequest",
    "sourceNode": "https://barcodes.example.net"
}
```

**Example Peer Approve Payload**

```json
{
    "action": "PeerApprove",
    "sourceNode": "https://barcodes.example.com"
}
```

**Example Peer Reject Payload**

```json
{
    "action": "PeerReject",
    "sourceNode": "https://barcodes.example.com",
    "message": "Peering rejected. This node does not add additional peers."
}
```

**Example Peer Pending Payload**

```
{
    "action": "PeerPending",
    "sourceNode": "https://barcodes.example.com",
    "message": "Your request has been received and is pending manual approval by the node operator"
}
```

### 4.4. Data Validation

A receiving Node MUST perform a strict, multi-step validation process before accepting data.

1. Announcement Validation: The incoming payload MUST be a valid Announcement object with a known type.
2. Record Validation: For Create or Update types, the record property MUST be a valid Record object.
3. Schema Fetching: The receiving node MUST fetch the JSON Schema from the URI in metadataSchema. Nodes SHOULD cache
   schemas to reduce latency and handle network failures. If a schema cannot be fetched, the record MUST be rejected.
4. Metadata Conformance: The metadata object within the record MUST successfully validate against the fetched schema.

If any validation step fails, the Node MUST NOT store or propagate the record. It SHOULD log the failure for
administrative review.

### 4.5. Federation Flow

Once peered, a source Node sends Create or Update Announcements to its peer's announcementEndpoint.

- Method: The HTTP request MUST be a POST request.

- Headers:
    - Content-Type MUST be application/json.
- Responses:
    - 202 Accepted: The announcement was received and queued for processing.
    - 400 Bad Request: The announcement payload was malformed or failed validation.
    - 401 Unauthorized / 403 Forbidden: Authentication or authorization failed.

---

### 4.6 Automated Peering Handshake

This sequence allows nodes to establish peering relationships with minimal manual intervention.

1. Initiation: Node A discovers Node B and sends a PeerRequest to its announcementEndpoint.
2. Response: Node B receives the request and acts based on its policy:
    - Auto-Approve: Node B adds Node A as a peer and sends a PeerApprove back to Node A.
    - Manual-Approve: Node B adds the request to a pending queue and sends a PeerPending back to Node A.
    - Auto-Reject: Node B sends a PeerReject back to Node A.
3. Completion (Manual Flow): If Node B's operator later approves the request, Node B sends a PeerApprove to Node A.
4. Reciprocation: Upon receiving PeerApprove, Node A adds Node B as a peer. For a bi-directional data flow, Node A
   SHOULD then initiate its own PeerRequest back to Node B, which Node B SHOULD auto-approve, having already established
   a relationship.

## 5. Security Considerations

- Authentication: Communication between Nodes SHOULD be authenticated using API keys exchanged during the peering
  handshake.
- Auto-Approval Risk: Automatic approval SHOULD be used with caution, ideally in conjunction with a domain allowlist.
- Rate Limiting: A Node MUST rate-limit incoming requests to all its public endpoints to prevent denial-of-service
  attacks.
- Transport Security: All communication MUST use HTTPS.

---

## 6. Privacy Considerations

- Public Peer Lists: The peersEndpoint is optional. Operators who require privacy for their peering arrangements SHOULD
  NOT include this property in their discovery file.
- Data Scope: The data shared via SBFP is intended to be public. No personally identifiable information (PII) SHALL be
  included in Record objects.
